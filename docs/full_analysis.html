SECTION 1

What We Built

Starting from the one-page proposal, we built a complete differentiable constraint programming solver — a working research prototype demonstrating that gradient-based methods can handle combinatorial satisfaction problems at practical scale.

Package

diffcp/ Python package — 7 modules, 5,000+ lines across solver, rounding, constraint, and model layers.

API

CP-SAT compatible interface: drop-in replacement for the OR-Tools subset needed by the benchmark suite.

Three Solver Architectures

  1. Standard Sinkhorn — baseline relaxation with Hungarian rounding. Fast, correct through N=32.
  2. Iterative Rounding — fix most-confident variable, re-optimize remainder. Extends to N=64 at the cost of wall-clock time.
  3. Augmented Lagrangian (ALM) — learnable dual variables that adaptively enforce violated constraints. Closes the gap through N=512.

Four Rounding Strategies

  • Hungarian — classical optimal assignment; deterministic, O(N³)
  • Gumbel-Sinkhorn — stochastic rounding with temperature-scaled noise
  • Greedy repair — random-swap local search with min-conflicts acceptance
  • Hybrid — ALM to near-feasibility, greedy repair to close the final gap

Benchmark Coverage

All three solvers benchmarked on N-Queens instances from N=8 through N=512, with CPU vs MPS (Apple Silicon GPU) profiling at each scale point.

SECTION 2

The Core Algorithm: Sinkhorn AllDifferent

The key insight is that an AllDifferent(x₁,…,xN) constraint over the domain {1,…,N} is equivalent to requiring that the assignment matrix is a permutation matrix. This opens a clean relaxation path.

The Mathematical Chain

Step 1 — AllDifferent = Permutation Matrix. Encode each variable xᵢ as a one-hot row vector. The N×N assignment matrix P must be a permutation matrix: exactly one 1 per row and per column.

AllDifferent(x₁,…,xN) ⟺ P ∈ Perm(N)

Step 2 — Birkhoff-von Neumann Relaxation. The Birkhoff-von Neumann theorem states that the set of doubly stochastic matrices (DS(N)) is the convex hull of the permutation matrices. We relax the integer constraint to the continuous polytope.

Perm(N) ⊂ DS(N) = conv(Perm(N))
P ∈ DS(N) ⟺ Pᵢⱼ ≥ 0, ΣⱼPᵢⱼ = 1, ΣᵢPᵢⱼ = 1

Step 3 — Sinkhorn Normalization. Given any non-negative matrix M, alternating row and column normalization converges to a doubly stochastic matrix. In log-space this is alternating logsumexp subtraction — a pair of batched matrix operations ideal for Tensor Cores.

R = log M
R ← R − logsumexp(R, dim=1, keepdim=True) // row normalize
R ← R − logsumexp(R, dim=0, keepdim=True) // col normalize
P = exp(R)

Step 4 — Temperature Annealing. We sharpen the doubly stochastic matrix toward a permutation matrix by scaling logits by 1/τ as τ → 0. High temperature = exploration; low temperature = commitment to discrete assignment.

P(τ) = Sinkhorn(M / τ, n_iter) τ: τ_start → τ_end (annealing schedule)

Step 5 — Hungarian Rounding. After optimization in the continuous relaxation, extract a valid permutation via the Hungarian algorithm (scipy.optimize.linear_sum_assignment), treating -P as a cost matrix.

Why Tensor Cores Love This

Each full Sinkhorn iteration reduces to exactly two matrix operations: a logsumexp along rows and a logsumexp along columns. For an N×N matrix these are O(N²) operations that tile perfectly onto Tensor Core systolic arrays. The gradient (for learning the cost matrix) flows through both normalizations via standard autograd — no custom CUDA kernels required.

SECTION 3

The Rounding Gap — The Core Problem

Key Insight

The continuous loss always converges to zero. The rounding step is where NP-hardness reasserts itself. Understanding this separation is the central theoretical contribution of this work.

What Works

Standard Sinkhorn + Hungarian rounding works cleanly for N ≤ 32. At these sizes, the doubly stochastic matrix after annealing is already "nearly permutation" — the largest entry per row exceeds 0.95 — and Hungarian rounding snaps it to a valid assignment with zero violations.

Where It Breaks

At N ≥ 64, the peak per-row confidence drops to the 0.6–0.8 range even after aggressive annealing. Hungarian rounding must resolve ambiguity across many rows simultaneously, and small errors compound. A conflict in row 47 forces a suboptimal choice in row 63, which cascades. The result: 6–8 violated AllDifferent constraints that no amount of re-running Sinkhorn will fix because the continuous optimum is not the discrete optimum.

The Formal Gap

minP ∈ DS(N) L(P) = 0     (continuous problem: always solvable)

minP ∈ Perm(N) L(P) = 0     (discrete problem: NP-hard in general)

Gap = L(Round(P*)) > 0        (rounding loss at continuous optimum)

This is precisely the integrality gap from LP relaxations in integer programming. The Sinkhorn relaxation is as tight as the LP relaxation for assignment problems — but for N-Queens with diagonal constraints, the LP gap is nonzero for large N.

Why This Is Interesting

The gap is not a failure of the gradient method — the gradient method is doing exactly what it should. The gap is a fundamental property of the constraint structure. The three attacks in the next section each take a different approach to closing it: iterative refinement, penalty adaptation, and local search repair.

SECTION 4

Three Attacks on the Rounding Gap

4a — Iterative Rounding

The simplest attack: after each Sinkhorn optimization, identify the variable with the highest assignment confidence (largest max-row-value in P), fix it to its argmax, remove it from the problem, and re-optimize the remaining N−1 × N−1 subproblem. Repeat until all variables are fixed.

Algorithm
  1. Run Sinkhorn optimization to convergence on full N×N problem
  2. Find i* = argmax_i (max_j P_ij) — the row with highest confidence
  3. Fix x_i* = argmax_j P_i*j; remove row i* and column j* from P
  4. Repeat on (N−1)×(N−1) subproblem until all fixed

Result: Solves N=64 in 216 seconds. Each of the N rounding steps requires a full Sinkhorn re-optimization on a shrinking subproblem. This is O(N) rounds each with O(N²·k) cost = O(N³·k) total — correct but impractical for large N.

4b — Augmented Lagrangian Method (the Breakthrough)

The key insight: instead of fixing variables one at a time, augment the loss with penalty terms for violated constraints and use dual variables that learn which constraints need enforcement.

The Augmented Lagrangian

LALM(P, μ) = L(P) + Σk μk · ck(P) + (ρ/2) · Σk ck(P)²

Where ck(P) is the violation of constraint k (AllDifferent row, AllDifferent column, diagonal), μk is the dual variable (Lagrange multiplier), and ρ is the penalty coefficient.

Dual Variable Update Rule

μk ← μk + ρ · ck(P*)

After each inner optimization, the dual variable for each violated constraint increases in proportion to the violation magnitude. On the next inner optimization, that constraint's penalty is correspondingly larger — the method automatically focuses gradient pressure on the hardest-to-satisfy constraints.

ρ Schedule

ρ grows geometrically across outer iterations: ρ_init = 5, multiplied by 2 each outer iteration up to ρ_max = 200. This prevents constraint penalties from overwhelming the objective function too early, while ensuring that as the solution refines, constraint satisfaction takes priority.

Results

The dual variables consistently converge in 2–3 outer iterations for N ≤ 256. ALM solves N=64 in 2.5 seconds (vs. 216s for iterative rounding), N=128 in 28s, N=256 in 176s — all with zero violations.

4c — Hybrid ALM + Greedy Repair (for N=512)

ALM drives violations from 29 down to 6 over 10 outer iterations but plateaus: once ρ hits ρ_max, dual variable updates produce diminishing returns. The remaining 6 violated constraints are in complex conflict cycles that gradient descent alone cannot resolve.

Greedy repair: After ALM reaches its plateau, we switch to a local search phase. Randomly sample a violated row, try all possible column swaps with non-violated rows, accept any swap that reduces the total violation count (min-conflicts heuristic). Repeat until zero violations.

Why This Works

The 6 remaining violations are not a gradient problem — they are a combinatorial local-search problem. The continuous solver has already done the heavy lifting; we just need a few targeted swaps to find the nearest valid permutation in discrete space.

Total time for N=512: ~34 minutes on M3 Ultra. Most of that is the 10 ALM outer iterations (each running 500 inner Sinkhorn steps at 512×512). The repair phase itself takes under a minute.

SECTION 7

Reproducing Our Results

Full setup and replication instructions are in REPRODUCING.md at the repo root. Quick start:

# Clone and set up environment
git clone https://github.com/ricku777-bear/diffcp.git
cd diffcp
python3 -m venv .venv
source .venv/bin/activate

# Install dependencies
pip install torch numpy scipy ortools

# Quick smoke test (N=8, should solve in under 1s)
python3 examples/nqueens_demo.py

# Full benchmark suite (all N, all solvers; takes ~1hr on M-series Mac)
python3 benchmarks/bench_rounding_gap.py

# ALM only, specific N (faster for experimentation)
python3 benchmarks/bench_rounding_gap.py --solver alm --n 128

# GPU vs CPU timing comparison
python3 benchmarks/bench_gpu.py
GPU Note

On Apple Silicon (M1–M4), PyTorch automatically uses MPS when device="mps" is set. On NVIDIA hardware, change this to device="cuda" — no other code changes required. The benchmark script auto-detects the available backend and falls back to CPU if neither MPS nor CUDA is present.

Package Architecture

The diffcp/ package is organized around three layers: constraint definitions (differentiable loss functions), solver engines (continuous optimization), and rounding strategies (discrete extraction). The CP-SAT compatible model layer sits on top of all three.

diffcp/
├── __init__.py # Package exports: DiffCPSolver, CpModel, CpSolver, ALMSolver
│
├── constraints.py # 7 differentiable constraint types:
│ # AllDifferentLoss, RowSumLoss, ColSumLoss,
│ # DiagonalConstraintVectorized,
│ # GraphColoringLossVectorized,
│ # JobShopLoss, SchedulingLoss
│
├── solver.py # Core DiffCPSolver engine: Sinkhorn loop, temperature
│ # annealing, loss computation, gradient update
│
├── rounding.py # 4 rounding strategies: hungarian(), gumbel_sinkhorn(),
│ # greedy_repair(), hybrid_round()
│
├── model.py # CP-SAT compatible API: CpModel, CpSolver, IntVar —
│ # drop-in for the OR-Tools subset used by benchmarks
│
├── iterative_solver.py # Iterative rounding approach: fix most-confident variable,
│ # re-optimize subproblem, repeat until done
│
└── augmented_lagrangian.py # ALM solver: dual variables μ, penalty schedule ρ,
# outer/inner loop, violation tracking, plateau detection

Key Design Decisions

  • Pure PyTorch, no custom CUDA kernels. All Sinkhorn operations are expressible as standard tensor ops — logsumexp, matmul, scatter. This means the code runs on any PyTorch backend (CPU, MPS, CUDA) with a single device flag change.
  • Constraint as loss function. Every constraint type implements a differentiable scalar loss. The solver doesn't know about constraint semantics — it just minimizes the sum of losses. Adding a new constraint type requires only implementing the loss function.
  • ALM outer/inner loop separation. The inner loop (Sinkhorn + gradient descent) is identical to the standard solver. ALM wraps it, updating dual variables and ρ between inner runs. This composability is why extending to new constraint types is straightforward.
